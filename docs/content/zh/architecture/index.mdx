## 概述

sast-link-next 是一个 Next.js 16 应用，可在两种模式下运行：

- **Web 模式**（`pnpm dev` / `pnpm build`）：通过 HTTP 提供服务的标准 Next.js 应用。
- **桌面模式**（`pnpm tauri dev` / `pnpm tauri build`）：相同的静态产物由 Tauri 2 WebView 壳加载，打包为 Windows、macOS 和 Linux 的原生安装包。

两种模式共用同一套代码库，唯一的运行时差异是 HTML/JS/CSS 资源的提供方式。

## 层级结构

```
浏览器 / WebView
    |
Next.js App Router（app/）
    |
    +-- Providers（SWR、ThemeProvider、MSW）
    |
    +-- 路由布局与页面
    |
    +-- 组件（components/）
    |
    +-- Hooks（hooks/）
    |
    +-- 状态（store/）-------- Zustand（仅客户端，同步）
    |
    +-- 数据（lib/api/）------ Axios + SWR（异步，面向后端）
    |
    +-- 校验（lib/validations/）- Zod + React Hook Form
    |
后端 API（NEXT_PUBLIC_API_BASE_URL）
```

## 关键设计决策

### 静态导出

`next.config.ts` 设置了 `output: "export"`，这意味着：

- 每个页面在构建时都会被预渲染为静态 HTML 文件。
- 运行时没有 Next.js 服务进程，所有产物均为静态文件。
- 不使用需要 Node.js 服务器的功能（如带运行时状态的 Route Handlers、Server Actions）。
- 必须设置 `images.unoptimized: true`，因为内置的图片优化 API 是服务端功能。

这一限制源于 Tauri 在生产环境中从 `out/` 目录以本地文件方式加载应用。

### 路由组

两个路由组分别处理未登录和已登录的关注点：

- `(tourist)/` — 登录、注册、重置、OAuth 回调。布局中无认证检查。
- `(user)/` — `/home` 及嵌套页面。布局会在 localStorage 中无 Token 时重定向至 `/login`。

### 状态管理分层

客户端状态由两种机制分别处理，各有不同特性：

| 关注点 | 机制 | 持久化 |
|---|---|---|
| 认证身份、UI 面板状态、账号列表 | Zustand | localStorage（via `persist` 中间件） |
| 服务端数据（用户资料） | SWR + Zustand | SWR 缓存（内存），拉取后写入 Store |

SWR 仅用于资料拉取（`useFetchProfile`）。响应写入 Zustand Store 后，整个应用中的组件可以同步读取资料，无需直接订阅 SWR。

### MSW 用于测试与本地开发

所有 API 调用均通过 `lib/api/client.ts` 中的 Axios 客户端进行。MSW 在网络层拦截这些调用，实现：

- 无需真实后端即可进行完整集成测试。
- 后端不可用时，设置 `NEXT_PUBLIC_API_MOCKING=true` 进行本地开发。

`mocks/handlers/` 中的处理器维护内存状态（票据、用户），跨请求序列模拟真实后端的行为。
