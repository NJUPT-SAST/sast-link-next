---

## 概述

密码重置遵循与注册相同的四步结构，主要区别如下：

- 通过已有用户名或邮箱查找账号（而非新注册学号）。
- 签发不同类型的票据（`resetPwdTicket`）。
- 后续请求使用不同的请求头（`RESETPWD-TICKET`）。
- 最终 API 调用为 `resetPassword` 而非 `userRegister`。

```
第一步：输入账号  →  第二步：验证邮箱  →  第三步：设置新密码  →  第四步：重置成功
```

## 第一步：账号查询

用户输入用户名或邮箱。提交后：

1. `verifyResetAccount(username)` 调用 `GET /verify/account?username=...&flag=2`。
2. 成功后返回 `resetPwdTicket`。
3. `sendMail(ticket, "reset")` 使用 `RESETPWD-TICKET` 请求头发送验证邮件。
4. 票据存储在组件状态中，流程进入第二步。

此处复用 `loginAccountSchema` 校验器（9 位学号或邮箱）。

## 第二步：邮箱验证

与注册第二步行为相同，区别在于：
- 使用 `RESETPWD-TICKET` 请求头替代 `REGISTER-TICKET`。
- 使用 `"reset"` 类型参数调用 `verifyCaptcha(ticket, captcha, "reset")`。

60 秒重发冷却与 `"S-"` 验证码前缀保持一致。

## 第三步：设置新密码

用户设置新密码，规则与注册相同：

- 最少 8 个字符。
- 须包含至少一个字母和一个数字。
- 确认密码字段必须匹配。

提交后，`resetPassword(password, resetTicket)` 向 `POST /user/resetPassword` 发送请求：
- 请求体字段名：`newPassword`（与注册使用的 `password` 不同）。
- 请求头：`RESETPWD-TICKET`。

## 第四步：重置成功

与注册相同的动画成功页面，包含指向 `/login` 的链接。

## API 差异对比

| 方面 | 注册 | 密码重置 |
|---|---|---|
| 验证端点 flag | `flag=0` | `flag=2` |
| 票据字段名 | `registerTicket` | `resetPwdTicket` |
| 请求头名称 | `REGISTER-TICKET` | `RESETPWD-TICKET` |
| 最终端点 | `POST /user/register` | `POST /user/resetPassword` |
| 最终请求体字段 | `password` | `newPassword` |
