## Overview

Login is a two-step process. The first step verifies that the account exists and issues a short-lived ticket. The second step uses that ticket together with the user's password to obtain a session token.

This design means the password is never sent in the first request, and the backend can attach rate limiting and brute-force protection to each ticket independently.

## Step 1 — Account verification

The user enters their username or email address. On submission, the frontend calls `verifyLoginAccount(username)`, which hits `GET /verify/account?username=...&flag=1`.

A successful response returns a `loginTicket` string. The ticket is stored in `useAuthStore` via `setLoginTicket(ticket)` and the flow advances to step 2.

```
User enters username/email
        |
verifyLoginAccount(username)
        |
GET /verify/account?flag=1
        |
{ Success: true, Data: { loginTicket: "..." } }
        |
useAuthStore.setLoginTicket(ticket)
        |
Advance to Step 2
```

If the account does not exist or the backend returns an error, an inline error message is shown and the flow stays on step 1.

### OAuth quick access

Step 1 also renders the `OtherLoginList` component, which shows OAuth provider buttons (GitHub, Feishu, QQ, Microsoft). Clicking a provider button redirects the user to the provider's authorization page. After authorization the provider redirects back to `/callback/feishu` or `/callback/github`.

## Step 2 — Password and token exchange

The user enters their password. On submission, the frontend calls `userLogin(password, loginTicket, oauthTicket?)`, which posts to `/user/login` with:

- `password` in the form body.
- `LOGIN-TICKET` header set to the ticket from step 1.
- `OAUTH-TICKET` header (optional) if the user arrived via an OAuth binding flow.

A successful response returns a `loginToken`. The token is persisted to localStorage via `setToken(token)` from `lib/token.ts`.

```
User enters password
        |
userLogin(password, loginTicket)
        |
POST /user/login
  body: password
  headers: LOGIN-TICKET, (OAUTH-TICKET if present)
        |
{ Success: true, Data: { loginToken: "..." } }
        |
setToken(loginToken)          ← localStorage
useAuthStore.clearLoginState() ← clears ticket + redirect
        |
Navigate to /home (or stored redirect URL)
```

## Auth store interaction

`useAuthStore` (persisted to `"auth-store"` in localStorage) tracks the login flow state:

| Field | Set by | Cleared by |
|---|---|---|
| `loginTicket` | `setLoginTicket` after step 1 | `clearLoginState` after step 2, `logout` |
| `redirect` | `setRedirect` before navigating to `/login` | `clearLoginState` after step 2, `logout` |
| `currentUser` | `setCurrentUser` after profile fetch | `clearCurrentUser`, `logout` |

Only `currentUser` is written to localStorage by the `persist` middleware. The ticket and redirect are session-only (lost on page refresh), which is intentional — a stale ticket from a previous session should not be reused.

## Validation

Form input is validated with Zod before the API call is made:

```ts
// lib/validations/auth.ts
export const loginAccountSchema = z.union([
  z.string().regex(/^\d{9}$/, "Must be a 9-digit student ID"),
  z.string().email("Must be a valid email address"),
]);

export const loginPasswordSchema = z.string().min(1, "Password is required");
```

## Logout

Calling `useAuthStore.logout()`:
1. Calls `clearToken()` to remove the token from localStorage.
2. Resets `currentUser` to `{ username: "", email: "" }`.
3. Clears `loginTicket` and `redirect`.

The calling component is responsible for navigating to `/login` after logout. The API call to `userLogout()` can be made before or alongside the store reset.
