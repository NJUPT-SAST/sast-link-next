## Home page (/home)

The home page displays a summary of the authenticated user's profile and provides quick navigation to editing and settings pages.

### Layout structure

`app/(user)/home/layout.tsx` is the top-level authenticated layout. It:
1. Calls `useFetchProfile()` on mount to load profile data into the Zustand stores.
2. Renders the `TopBar` at the top.
3. Arranges the main content, `@infoPanel`, and `@profilePanel` in a responsive grid via parallel routes.

### Page content

`app/(user)/home/page.tsx` renders:

- **Profile header**: Avatar, nickname, department, and organization from `useUserProfileStore`.
- **Profile completion**: A visual indicator of how many optional fields are filled.
- **Quick actions**: Buttons linking to `/home/edit` (Edit Info) and `/home/edit/safety` (Safety Settings).
- **Bio and links**: The user's bio text and up to three custom links.
- **Common apps grid**: Fixed links to four integrated applications — Approval, OJ (Online Judge), Visual Arts, and Evento.

## Profile editing (/home/edit)

`app/(user)/home/edit/page.tsx` provides two sections:

### Avatar upload and cropping

The avatar editor uses `react-avatar-editor` to let users crop and zoom a profile picture before uploading.

- The user selects an image file via a file input.
- The editor renders a circular crop canvas.
- **Zoom**: The scroll wheel adjusts zoom from 1x (minimum, no zoom) to 5x (maximum).
- The user clicks "Save" to export the cropped image as a `Blob` and upload it via `uploadAvatar(blob)`.
- `uploadAvatar` posts to `POST /profile/uploadAvatar` using `multipart/form-data` with field name `avatarFile`.
- On success, the response contains `filePath` — the new avatar URL, which is then reflected in the profile store.

### Profile fields form

The form uses React Hook Form with Zod validation (`lib/validations/profile.ts`):

| Field | Constraint |
|---|---|
| Nickname | Maximum 20 characters |
| Bio | Maximum 200 characters |
| Link 1 | Valid URL or empty |
| Link 2 | Valid URL or empty |
| Link 3 | Valid URL or empty |

On submission, `editProfile(data)` posts to `POST /profile/changeProfile` with the updated fields. After a successful response, `useUserProfileStore.updateProfile(fields)` is called to keep the store in sync without a full refetch.

The `EditableProfileType` sent to the backend:

```ts
interface EditableProfileType {
  nickname?: string | null;
  org_id?: number;
  bio?: string;
  link?: string[] | null;
  hide?: string[] | null;
}
```

## Safety settings (/home/edit/safety)

`app/(user)/home/edit/safety/page.tsx` currently shows:

- **Email**: The user's registered email address (read-only). Sourced from `useUserProfileStore`.
- **Email rebinding**: A placeholder section. The actual rebinding flow is not yet implemented in the current version.

## Data flow for profile pages

```
useFetchProfile() (in /home layout)
        |
GET /profile/getProfile
        |
useUserProfileStore.setProfile(data)
useAuthStore.setCurrentUser({ username, email })
useUserListStore.updateAccount({ email, nickName, avatar })
        |
All profile pages read from useUserProfileStore
```

Because the profile is fetched once at the layout level and written to the store, all nested pages (`/home`, `/home/edit`, `/home/edit/safety`) can read profile data synchronously from `useUserProfileStore` without issuing additional API calls.
