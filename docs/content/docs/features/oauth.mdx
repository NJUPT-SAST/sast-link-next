## Supported providers

sast-link-next integrates with two OAuth providers for third-party login:

| Provider | Callback route | Backend endpoint |
|---|---|---|
| GitHub | `/callback/github` | `GET /login/github/callback` |
| Feishu (Lark) | `/callback/feishu` | `GET /login/lark/callback` |

The `OtherLoginList` component in `components/auth/other-login-list.tsx` renders buttons for GitHub, Feishu, QQ, and Microsoft. QQ and Microsoft are displayed but their flows are not implemented in the current version.

## OAuth authorize flow (PKCE)

When the application itself acts as an OAuth client (i.e., a third-party application requests authorization through SAST Link), the `oAuth()` function in `lib/api/oauth.ts` is used to construct the authorization URL and redirect the browser.

```ts
// lib/api/oauth.ts
export function oAuth(data: OAuthParams) {
  const params = new URLSearchParams();
  for (const [key, value] of Object.entries(data)) {
    if (value) params.append(key, value);
  }

  const token = getToken();
  if (token) {
    params.append("part", token);
  }

  window.location.href = `${API_BASE_URL}/oauth2/authorize?${params.toString()}`;
}
```

The `OAuthParams` interface accepts the standard PKCE parameters:

| Parameter | Description |
|---|---|
| `client_id` | OAuth application identifier |
| `code_challenge` | PKCE code challenge (S256) |
| `code_challenge_method` | Always `"S256"` for PKCE |
| `redirect_uri` | Where the authorization server redirects after approval |
| `response_type` | Always `"code"` for authorization code flow |
| `scope` | Requested permission scopes |
| `state` | Opaque value for CSRF protection |

If a session token is present, it is appended as the `part` parameter so the authorization server can associate the approval with the logged-in user.

## Social login callback flow

When a user clicks GitHub or Feishu login on the login page, the browser is redirected to the provider's authorization endpoint. After the user authorizes, the provider redirects back to the corresponding callback route with `code` and `state` query parameters.

The callback page calls the appropriate backend endpoint:

```ts
// GitHub
getGithubLoginStatus(code, state)
// → GET /login/github/callback?code=...&state=...

// Feishu
getFeishuLoginStatus(code, state)
// → GET /login/lark/callback?code=...&state=...
```

The backend exchanges the code for provider credentials and returns one of two responses:

1. **Direct login**: `{ loginToken: "..." }` — The user already has a SAST Link account linked to this provider. The token is stored and the user is redirected to `/home`.
2. **Account binding**: `{ oauthTicket: "..." }` — The provider account is not yet linked. The OAuth ticket is stored and the user is redirected to the login flow's step 2 (password entry). After entering their password, `userLogin` is called with both `loginTicket` and `oauthTicket`, which the backend uses to link the provider account to the SAST Link account.

## Bind status

`getUserBindStatus()` calls `GET /profile/bindStatus` and returns a `string[]` of provider identifiers that are currently linked to the authenticated user's account. This is displayed in the profile area via the `BindAppItem` component.
