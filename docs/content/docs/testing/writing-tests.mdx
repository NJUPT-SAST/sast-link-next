## MSW handler patterns

The mock handlers in `mocks/handlers/` maintain in-memory state that mirrors the real backend's session model. Understanding this state is important for writing integration tests.

### Auth handler state

`mocks/handlers/auth.ts` tracks:
- A `tickets` map from ticket string to `{ username, verified, type }`.
- Flags on each ticket for whether the captcha has been verified.

When a test calls `verifyRegistAccount("B21041234")`, the handler issues a ticket and stores it. Subsequent calls to `sendMail`, `verifyCaptcha`, and `userRegister` with that ticket are validated against this in-memory map.

### User handler state

`mocks/handlers/user.ts` tracks:
- A `users` map with pre-seeded test accounts.
- A `sessions` map from `loginToken` to `userId`.

The `Token` header on every authenticated request is looked up in `sessions`. If the token is absent or invalid, the handler returns an error response.

### Overriding handlers in a test

Use `server.use(...)` to add a one-off handler override for a specific test:

```ts
import { server } from "@/mocks/server";
import { http, HttpResponse } from "msw";

test("shows error when login fails", async () => {
  server.use(
    http.post("/user/login", () =>
      HttpResponse.json({ Success: false, ErrCode: 401, ErrMsg: "Invalid credentials", Data: null })
    )
  );

  // ... render and interact
});
```

`server.resetHandlers()` in `afterEach` (set up in `jest.setup.ts`) removes this override after the test completes.

## Rendering components

Use `render` from `@testing-library/react`. Components that use Zustand stores or SWR do not require a custom provider — both work without a wrapping context in tests.

```tsx
import { render, screen } from "@testing-library/react";
import { AuthShell } from "@/components/auth/auth-shell";

test("renders heading", () => {
  render(<AuthShell heading="Sign in" description="Enter your credentials" />);
  expect(screen.getByText("Sign in")).toBeInTheDocument();
});
```

### Components that use Next.js router

`next/navigation` is mocked globally in `jest.setup.ts`. The mock returns empty strings for `usePathname` and a no-op object for `useRouter`. If a test needs specific router behavior, mock the module locally:

```ts
import * as navigation from "next/navigation";
jest.spyOn(navigation, "useRouter").mockReturnValue({ push: jest.fn() } as any);
```

## Interacting with forms

Use `@testing-library/user-event` for realistic form interaction (fires keyboard, pointer, and focus events in the correct sequence):

```tsx
import userEvent from "@testing-library/user-event";

test("submits login form", async () => {
  const user = userEvent.setup();
  render(<LoginStep1 />);

  await user.type(screen.getByLabelText("Username"), "B21041234");
  await user.click(screen.getByRole("button", { name: "Next" }));

  expect(await screen.findByText("Enter your password")).toBeInTheDocument();
});
```

## Testing Zustand stores

Zustand stores can be tested by importing and calling them directly, without rendering a component:

```ts
import { useAuthStore } from "@/store/use-auth-store";

test("logout clears user", () => {
  const store = useAuthStore.getState();
  store.setCurrentUser({ username: "Alice", email: "alice@example.com" });
  store.logout();
  expect(useAuthStore.getState().currentUser.username).toBe("");
});
```

Reset store state between tests to prevent leakage:

```ts
afterEach(() => {
  useAuthStore.setState({ currentUser: { username: "", email: "" }, loginTicket: null, redirect: null });
});
```

## Testing API functions

API functions can be tested by calling them directly. MSW intercepts the Axios requests:

```ts
import { verifyLoginAccount } from "@/lib/api/auth";

test("returns loginTicket for existing user", async () => {
  const res = await verifyLoginAccount("B21041234");
  expect(res.data.Success).toBe(true);
  if (res.data.Success) {
    expect(res.data.Data.loginTicket).toBeDefined();
  }
});
```

Because `jest.setup.ts` starts the MSW server before all tests, no additional setup is needed to intercept requests in API-level tests.
