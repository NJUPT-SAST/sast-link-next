## Summary

sast-link-next is a Next.js 16 application that runs in two modes:

- **Web mode** (`pnpm dev` / `pnpm build`): a standard Next.js application served over HTTP.
- **Desktop mode** (`pnpm tauri dev` / `pnpm tauri build`): the same static export loaded by a Tauri 2 WebView shell, producing native installers for Windows, macOS, and Linux.

Both modes share an identical codebase. The only runtime difference is how the HTML/JS/CSS assets are served.

## Layer structure

```
Browser / WebView
    |
Next.js App Router (app/)
    |
    +-- Providers (SWR, ThemeProvider, MSW)
    |
    +-- Route layouts and pages
    |
    +-- Components (components/)
    |
    +-- Hooks (hooks/)
    |
    +-- State (store/) ---- Zustand (client-only, synchronous)
    |
    +-- Data (lib/api/) --- Axios + SWR (async, server-facing)
    |
    +-- Validation (lib/validations/) --- Zod + React Hook Form
    |
Backend API (NEXT_PUBLIC_API_BASE_URL)
```

## Key design decisions

### Static export

`next.config.ts` sets `output: "export"`. This means:

- Every page is pre-rendered to a static HTML file at build time.
- There is no Next.js server process at runtime; all assets are static files.
- Server-side features that require a running Node.js server (Route Handlers with runtime state, Server Actions with side effects beyond static generation) are not used.
- The `images.unoptimized: true` flag is required because the built-in Next.js Image Optimization API is a server-side feature.

This constraint exists because Tauri loads the application from `out/` as local files rather than from a server.

### Route groups

Two route groups separate unauthenticated and authenticated concerns:

- `(tourist)/` — login, register, reset, OAuth callbacks. No authentication check in the layout.
- `(user)/` — `/home` and nested pages. The layout redirects to `/login` if no token is present in localStorage.

### State management split

Client state is handled by two different mechanisms with different characteristics:

| Concern | Mechanism | Persistence |
|---|---|---|
| Authentication identity, UI panel state, account list | Zustand | localStorage (via `persist` middleware) |
| Server data (user profile) | SWR + Zustand | SWR cache (memory), stores updated on fetch |

SWR is used only for the profile fetch (`useFetchProfile`). The response is written into Zustand stores so that components throughout the application can read profile data synchronously without subscribing to SWR directly.

### MSW for testing and local development

All API calls go through the Axios client in `lib/api/client.ts`. MSW intercepts these calls at the network layer, allowing:

- Full integration tests without a real backend.
- Local development with `NEXT_PUBLIC_API_MOCKING=true` when the backend is unavailable.

The MSW handlers in `mocks/handlers/` maintain in-memory state (tickets, users) that mirrors the real backend's behavior across a request sequence.
