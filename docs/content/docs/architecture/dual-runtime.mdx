## Overview

sast-link-next targets two deployment environments from a single codebase:

- **Web**: served by any static file host or CDN.
- **Desktop**: bundled as a native application using Tauri 2.

Both use the same HTML/CSS/JS output produced by `pnpm build`.

## Static export configuration

`next.config.ts` sets `output: "export"`, which causes `pnpm build` to write all pages and assets to the `out/` directory as static files. No Node.js server process is needed at runtime.

```ts
// next.config.ts
const nextConfig: NextConfig = {
  output: "export",
  images: {
    unoptimized: true,
    remotePatterns: [
      {
        protocol: "https",
        hostname: "sast-link-1309205610.cos.ap-shanghai.myqcloud.com",
      },
    ],
  },
  assetPrefix: isProd ? undefined : `http://${internalHost}:3000`,
};
```

`images.unoptimized: true` is required because the Next.js Image Optimization API needs a server. Remote images are fetched directly from `sast-link-1309205610.cos.ap-shanghai.myqcloud.com` without server-side optimization.

`assetPrefix` is set in development to the Tauri dev host address so that the WebView can resolve assets correctly when it opens the local dev server.

## Tauri configuration

`src-tauri/tauri.conf.json` connects the Tauri shell to the Next.js output:

```json
{
  "build": {
    "beforeDevCommand": "pnpm dev",
    "beforeBuildCommand": "pnpm build",
    "frontendDist": "../out"
  }
}
```

- `beforeDevCommand`: When you run `pnpm tauri dev`, Tauri first starts `pnpm dev` to launch the Next.js development server, then opens the WebView pointing to `http://localhost:3000`.
- `beforeBuildCommand`: When you run `pnpm tauri build`, Tauri first runs `pnpm build` to generate `out/`, then bundles it into the installer.
- `frontendDist`: In production, Tauri loads the application from `../out` as local files using a custom protocol.

## Rust entry point

`src-tauri/src/lib.rs` contains the Tauri application setup. In debug builds, it enables the Tauri log plugin to surface Rust-level log output to the developer console:

```rust
#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::Builder::default()
        .plugin(tauri_plugin_log::Builder::new().build())  // debug only
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

## Platform support

The CI pipeline (`build-tauri.yml`) builds installers for four targets:

| Platform | Installer format |
|---|---|
| Windows | NSIS installer, MSI |
| macOS | DMG, app bundle |
| Linux | AppImage, DEB |
| macOS (ARM) | DMG, app bundle |

## Constraints imposed by static export

Because the application is a static export, the following Next.js features cannot be used:

- Route Handlers with runtime state (no `POST` endpoints that modify server memory)
- Server Actions (require a Node.js server process)
- Dynamic server-side rendering (`getServerSideProps` equivalent in App Router)
- Incremental Static Regeneration

All data fetching happens on the client side via Axios and SWR against the external backend API (`NEXT_PUBLIC_API_BASE_URL`).
