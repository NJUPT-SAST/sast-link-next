## Branch naming

All branches must use one of the following prefixes:

| Prefix | Use for |
|---|---|
| `feat/` | New features |
| `fix/` | Bug fixes |
| `docs/` | Documentation changes only |
| `test/` | Adding or updating tests |
| `refactor/` | Code restructuring without behavior change |
| `ci/` | CI/CD pipeline changes |
| `chore/` | Dependency updates, tooling, configuration |

Examples: `feat/oauth-qq-integration`, `fix/register-step2-resend-cooldown`, `docs/state-management`.

## Commit messages

Commits follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

```
<type>(<optional scope>): <short summary>

<optional body>

<optional footer>
```

**Type** must be one of: `feat`, `fix`, `docs`, `test`, `refactor`, `ci`, `chore`, `style`, `perf`.

**Scope** is optional and should identify the affected module or feature (e.g., `auth`, `profile`, `store`, `api`).

**Summary** should be in the imperative mood, lowercase, and not end with a period.

Examples:
```
feat(auth): add Feishu OAuth callback handler
fix(store): correct removeAccount filter return value
test(api): add coverage for verifyResetAccount
docs: document password reset flow
chore: upgrade Zustand to 5.0.12
```

## Pull request requirements

A PR must pass all three of the following checks before it can be merged:

1. **Lint**: `pnpm lint` must exit with code 0. No ESLint errors are permitted (warnings are acceptable).
2. **Tests**: `pnpm test:coverage` must pass including all coverage thresholds (branches 60%, functions 60%, lines 70%, statements 70%).
3. **Build**: `pnpm build` must complete without errors, producing a valid `out/` directory.

These are enforced by the `quality.yml` and `test.yml` GitHub Actions workflows on every pull request.

## Adding components

When adding a new shadcn/ui-style primitive:

```bash
pnpm dlx shadcn@latest add <component-name>
```

This generates a file in `components/ui/`. Review and adjust the generated code to match the project's Tailwind v4 CSS variable conventions before committing.

## Adding a new route

1. Create a folder under the appropriate route group (`(tourist)/` or `(user)/`).
2. Add `page.tsx`. If the page has private sub-components, place them in a `_components/` subdirectory.
3. Add a corresponding `page.test.tsx`.
4. If the route requires a new layout, add `layout.tsx`.
5. Update this documentation if the route introduces a new user-facing feature.

## Environment variables

Never commit `.env.local` or any file containing real credentials. The repository `.gitignore` excludes `.env*.local`. Use `.env.example` (if present) to document required variables without values.
