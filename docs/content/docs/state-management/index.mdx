## Overview

Client-side state is managed with [Zustand 5](https://zustand.docs.pmnd.rs/). There are four stores, each with a distinct responsibility:

| Store | File | Persisted | Purpose |
|---|---|---|---|
| `useAuthStore` | `store/use-auth-store.ts` | Yes (partial) | Authentication identity and login flow state |
| `useUserListStore` | `store/use-user-list-store.ts` | Yes (full) | Remembered account list for the account switcher |
| `useUserProfileStore` | `store/use-user-profile-store.ts` | No | Full profile data for authenticated pages |
| `usePanelStore` | `store/use-panel-store.ts` | No | UI state for the home page side panel |

## useAuthStore

Tracks the currently identified user and the transient login flow state.

```ts
interface AuthState {
  currentUser: CurrentUserProfile;  // { username, email, userId?, avatar?, ... }
  loginTicket: string | null;       // ticket from verifyLoginAccount; cleared after login
  redirect: string | null;          // URL to navigate to after successful login

  setCurrentUser(user: CurrentUserProfile): void;
  clearCurrentUser(): void;
  setLoginTicket(ticket: string): void;
  setRedirect(redirect: string): void;
  clearLoginState(): void;          // clears loginTicket + redirect
  logout(): void;                   // clearToken() + resets all fields
}
```

**Persistence**: The `persist` middleware writes only `currentUser` to localStorage under the key `"auth-store"`. `loginTicket` and `redirect` are in-memory only — they are not restored on page reload.

**Logout behavior**: `logout()` calls `clearToken()` (removes the session token from localStorage) and then resets `currentUser`, `loginTicket`, and `redirect` to their initial values in a single `set` call.

## useUserListStore

Maintains a list of accounts that have been logged into on this device, enabling the account switcher on the root page.

```ts
interface UserListState {
  accounts: UserAccount[];          // { token, avatar, nickName, email, userId }

  addAccount(account: UserAccount): void;     // no-op if userId already exists
  updateAccount(update: { email: string; nickName: string; avatar: string | null }): void;
  removeAccount(identifier: number | string): void; // by array index or email
}
```

**Persistence**: The entire `accounts` array is persisted to localStorage under the key `"user-list-store"`.

**Deduplication**: `addAccount` checks `userId` before appending. If an account with the same `userId` already exists, the call is a no-op. Use `updateAccount` to refresh display name and avatar for an existing entry.

**Removal**: `removeAccount` accepts either a numeric array index or an email string. The email-based variant is used when the exact position in the array is not known.

## useUserProfileStore

Holds the full `UserProfileType` returned by `GET /profile/getProfile`. Used by all authenticated pages to read profile data without additional API calls.

```ts
interface UserProfileState {
  profile: UserProfileType;

  setProfile(profile: UserProfileType): void;      // replaces entire profile
  updateProfile(fields: EditableFields): void;     // merges partial fields
  resetProfile(): void;                            // restores initial placeholder values
}
```

**Initial value**: All string fields are set to `"~"` and nullable fields are set to `null`. This provides a recognizable placeholder state rather than empty strings.

**Not persisted**: Profile data is always re-fetched from the API on mount (via `useFetchProfile`). Persisting it would risk serving stale data.

**`updateProfile`**: Merges the supplied `EditableFields` into the current profile using object spread. Used by the edit page after a successful `editProfile` API call to avoid a full refetch.

## usePanelStore

Simple UI state store for controlling the info side panel on the home page.

```ts
interface PanelState {
  homeInfoPanel: boolean;
  setHomeInfoPanel(open: boolean): void;
}
```

Not persisted. The panel defaults to closed (`false`) on every page load.

## Data flow example

The following shows how the stores are populated after a successful login:

```
1. userLogin() returns loginToken
        |
2. setToken(loginToken)          → localStorage
3. useAuthStore.clearLoginState()
        |
4. Navigate to /home
        |
5. useFetchProfile() (in home layout)
        |
6. GET /profile/getProfile
        |
7. useUserProfileStore.setProfile(data)
8. useAuthStore.setCurrentUser({ username: data.nickname, email: data.email })
9. useUserListStore.updateAccount({ email, nickName, avatar })
```
